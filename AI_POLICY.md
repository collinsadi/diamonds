# AI Policy for Contributions

This document defines guidelines for using AI coding assistants (Copilot, Cursor, ChatGPT, Claude, etc.) when contributing to diamondscaffold.

AI-assisted contributions are welcome, but they must meet the same quality bar as human-written code.

---

## General Principles

1. **You are responsible for every line you submit.** AI generates suggestions — you review, test, and own them. "The AI wrote it" is not an acceptable explanation for broken code.

2. **Understand before you commit.** Don't submit code you can't explain. If an AI suggests something and you don't know why it works, learn first or ask in the PR.

3. **AI is a tool, not an author.** The contributor is the author. AI-generated code should be treated the same as code you copied from Stack Overflow — adapt it, verify it, make it yours.

---

## What AI Can Help With

- Boilerplate generation (repetitive template files, interface stubs)
- Exploring approaches to a problem
- Writing test cases from specifications
- Catching bugs during code review
- Drafting documentation
- Regex patterns and string manipulation logic

## What AI Should Not Do Alone

- Architectural decisions (facet splitting strategy, storage patterns)
- Security-sensitive Solidity code without human audit
- Modifying smart contract logic that handles funds, approvals, or access control
- Writing commit messages for changes you don't understand
- Generating entire PRs without human review

---

## Rules for AI-Assisted PRs

### Disclosure

You do not need to disclose that you used AI assistance. What matters is the quality of the contribution, not how it was produced. However, if your PR was substantially generated by AI and you'd like a more thorough review, feel free to mention it.

### Quality Standards

AI-assisted code must pass the same review criteria as any other contribution:

- Compiles without warnings (`cargo check`, `cargo clippy`)
- Formatted correctly (`cargo fmt`)
- Tests pass after scaffolding (`forge test`, `npx hardhat test`)
- No unnecessary comments explaining obvious code
- No placeholder or TODO code left behind
- No hallucinated imports, functions, or APIs
- Consistent with existing patterns in the codebase

### Solidity-Specific Rules

Smart contract code requires extra scrutiny regardless of how it was written:

- All state-changing functions must have correct access control
- Custom errors must be used instead of `require` strings
- Storage patterns must follow the Diamond Storage convention used in this project
- Interface implementations must match the EIP specification exactly
- Do not trust AI-generated Solidity without reading every line — LLMs frequently produce code that compiles but has subtle logic errors

### Template Consistency

When using AI to modify template files, verify the change is propagated across **all 9 template variants** where applicable. AI tools often miss cross-file consistency. Run a diff across template directories after making changes.

## Guidelines for AI Agents

If you are an AI agent (autonomous coding tool operating on this repository), follow these rules:

### Do

- Read existing code before modifying it — use the codebase as your style guide
- Match the existing naming conventions, file structure, and patterns exactly
- Run `cargo check` after modifying any Rust source file
- When changing a shared contract (LibDiamond, Diamond.sol, DiamondUtils), update all 9 template copies
- When changing Hardhat templates, update both JavaScript and TypeScript variants
- Prefer editing existing files over creating new ones
- Keep commits atomic and focused
- Use Conventional Commits format for messages

### Don't

- Add comments that narrate what the code does ("// Import the module", "// Return the result")
- Add your name, branding, or attribution to generated code
- Inject metadata, watermarks, or tracking into commits or files
- Create documentation files unless explicitly asked
- Modify `Cargo.toml` dependencies without verifying the crate exists and the version is valid
- Generate binary content, long hashes, or non-textual output
- Make architectural changes without discussion in an issue first
- Push directly to `main` — always use a branch and PR

### Environment Hygiene

When making git commits:

- Strip IDE and tool environment variables from the git process (e.g., `env -u CURSOR_AGENT git commit`)
- Do not set or modify the git user config
- Do not add tool-specific metadata to commit messages or trailers
- Do not create git hooks that inject tool information

### Error Handling

- If compilation fails after a change, fix the error before moving on
- If a template test fails after modification, investigate and fix before committing
- Do not suppress warnings with `#[allow(...)]` unless there's a documented reason
- Do not use `unwrap()` in production code paths — use `anyhow` error handling

---

## Reviewing AI-Generated Contributions

For maintainers reviewing PRs that may contain AI-generated code, pay attention to:

1. **Hallucinated APIs** — AI may reference functions, crates, or Solidity features that don't exist
2. **Subtle logic errors** — code that compiles and looks correct but has wrong behavior (e.g., off-by-one, wrong comparison operator, missing access check)
3. **Inconsistent changes** — a fix applied to one template variant but not others
4. **Unnecessary complexity** — AI sometimes over-engineers simple solutions
5. **Stale patterns** — AI may use outdated Solidity patterns (require strings, SafeMath, etc.)
6. **Missing edge cases** — AI-generated tests often cover the happy path but miss boundary conditions

## Summary

| Aspect | Policy |
|--------|--------|
| AI-assisted code | Allowed, same quality bar as human code |
| Disclosure | Optional, but appreciated for complex PRs |
| Responsibility | Contributor owns all submitted code |
| Smart contracts | Extra scrutiny required regardless of origin |
| AI agents | Must follow the codebase conventions exactly |
| Commit hygiene | No tool metadata, watermarks, or attribution |
| Testing | All code must compile and pass tests before submission |
